ExtJS 4

Posted on 2013-10-10 | Post modified 2017-06-11 | In Frameworks

Ext JS 4 is Sencha’s latest JavaScript framework for developing cross-platform web applications.

  • User interface widgets
  • Data layer
  • MVC architecture
  • Theming
  • Charting

All Ext JS 4 classes inherit from a common base class, named Ext.Base

Constructors are special methods that are executed when a class is instantiated (either using the Ext.create(..) or new syntax) and are used to prepare the object in any way needed. For example, it could be used to set up default property values.

Creating new classes:

Call Ext.define with params for class name and a configuration object.

The Layout System

Component Layouts

  • used to lay out the internal items of components
  • all the standard components already have their corresponding component layout, so usually you don’t need detailed knowledge, it just works.

Container Layouts

  • A container is a widget that can contain child items.
  • Layouts are implemented by a container to visually organize (location and size) child items in its content body.
Auto Layout
  • The default layout for a container.
  • AutoLayout provides only a passthrough of any layout calls to any child containers.
  • A child’s width may extent to the whole container body if not constrained.
  • Doesn’t require any configuration, just add child items.
Anchor Layout

This is a layout that enables anchoring of contained elements relative to the container’s dimensions. This is perfect when you want to have child items dynamically resize to fit the container’s content body.

The anchor parameter, which can be specified on the container’s child items, is used to calculate the size of the child item relative to the parent’s content body size and is specified as a percentage, an offset, which is an integer. The anchor parameter is a string, using the following format:

anchor : "width, height" // or "width height"
Absolute Layout

The Absolute layout is perfect for when you need complete control over the positioning of the widget layout. It fixes the position of a child by setting the CSS “position” attribute of the child’s element to ‘absolute’ and sets the top and left attributes to the x and y parameters that you set on the child items.

Fit Layout

Use this for a container containing a single child item that you want to use all available space of its container.

Accordion Layout

The accordion layout is useful when you want to display multiple panels vertically stacked. It allows you to vertically stack items where all but one item is collapsed.

It’s important to note that the Accordion layout can function well only with a Ext.panel.Panel and two of its subclasses, Ext.Grid.Panel and Ext.Tree.Panel.

If you’d like a tabpanel inside an accordion layout wrap it in a panel.

Card Layout

The Card layout ensures that its children conform to the size of the Container. Unlike the Fit Layout, however, the Card Layout can have multiple children under its control.

Manages multiple child Components, each fitted to the Container, where only a single child Component can be visible at any given time. This layout style is most commonly used for wizards, tab implementations, etc.

To move between components call the setActiveItem() method of the container’s layout object.

Layout (the Container and Component Models)

Method doLayout() is a way to force recalculation of child items in the layout tree. It’s ambiguous and recursive so might not be necessary and takes time. Often triggers redundant calls to doLayout() of child components. When changing items in a container you might need to call the container’s doLayout(). Adding items to containers should automatically trigger layout. Passed arrays only triggers a single layout. This behavior can be manually overridden.

Ext 3 BoxLayout issues. You had to manually call doLayout(). This should be fixed in ExtJS4.

AutoLayout issues. Autolayout the roles are reversed so that child items need to tell it’s container what size to be.

In ExtJS 4 layouts will no longer alter a components sizing properties. If you try to change size of hidden components, their sizes will not change. The size will be set when the item becomes visible.

Container Layouts

Containers are:

Container
    Viewport
    ButtonGroup
    Panel, TabPanel
    Window
    ...

Container is the base class for any component that may contain other components.

Containers only perform perform sizing and positioning on its child Components if it’s configured with a layout.

Viewport

The Viewport does not provide scrolling, so child Panels within the Viewport should provide for scrolling if needed using the autoScroll config.

Layout Configurations

Similar to anchor layout but adds the ability for x/y positioning using the child component’s configs.

Border Layout

Child components should be configured with a region of (´north´, ´east´, ´south´, ´west´ or ´center´). The north and south region children should define a height and the west and east region children should define a width. Sizes can be given as number of pixels or as a string percentage. Alternatively you can use the flex configuration. The center region is mandatory and usually flexible taking up all available space left. The center region has a flex value of 1.

Child components can also be configured with split: true to create a splitter between regions, and collapsible: true to be able to collapse a region.

To add/remove Components within a BorderLayout, have them wrapped by an additional Container which is directly managed by the BorderLayout. If the region is to be collapsible, the Container used directly by the BorderLayout manager should be a Panel.

Column Layout

This is the layout style of choice for creating structural layouts in a multi-column format where the width of each column can be specified as a percentage or fixed width, but the height is allowed to vary based on the content.

Configure child components by using columnWidth configuration. The sum of column widths should be 1.

HBox/VBox Layout

A layout that arranges items horizontally/vertically across a Container. Use the flex configuration of the child items to divide up the space.

Table Layout

This layout allows you to easily render content into an HTML table. The total number of columns can be specified, and rowspan and colspan can be used to create complex layouts within the table.

Component Layouts

Component layouts define how a component sizes its child items. Component layouts are responsible for arranging the HTML for components

The GridPanel

The ExtJS 4 grid only renders the features that the grid uses to minimize the amount of generated html and improve performance.

A simple Grid

For a grid you need to declare what columns it contains, and give it a data source (store). A column’s dataIndex property contains the name of the store field that you want to display in that column.

Ext.create('Ext.grid.Panel', {
    height:  80,
    width:   300,
    title:   'A Simple Grid',
    renderTo: Ext.getBody(),

    columns: [{
        text: 'Books',
        flex: 1,
        sortable: false,
        dataIndex: 'book'
    }, {
        text: 'Author',
        width: 100,
        sortable: true,
        dataIndex: 'author'
    }],

    store: Ext.create('Ext.data.ArrayStore', {
        fields: [
            {name: 'book'},
            {name: 'author'}
        ],
        data: [['Ext JS 4: First Look', 'Loiane Groner']]
    })
});

The selModel config determines how you can select items in the grid. If you would like to have a separate column with checkboxes, you can set it like:

selModel: Ext.create('Ext.selection.CheckBoxModel')

Columns

The column classes are defined in Ext.grid.columns package.

String (default)

If no xtype attribute is specified the data in a column will be displayed as string.

Date

This columntype is for displaying date values. We can configure how we want the date formatted via the format property which follow the rules of PHP date formats.

{
    xtype: 'datecolumn',
    dataIndex: 'anyDate',
    format: 'Y-m-d'
}
Number

The numbercolumn type is used for displaying number values, money etc. We can use a renderer in combination with the Format function to manipulate how the number is formatted.

{
    xtype:     'numbercolumn',
    dataIndex: 'anyNumber',
    renderer:  Ext.util.Format.numberRenderer('0.000')
}

There’s lots of possibilities for formatting values. Take a look at Ext.util.Format

Boolean

The booleancolumn type by default displays true or false but can be configured via the trueText and falseText to display whatever strings we like for true and false values.

{
    xtype:     'booleancolumn',
    dataIndex: 'anyBool',
    trueText:  'Yes',
    falseText: 'No'
}
Template

This column type uses an XTemplate for displaying the column data. In this sample we’re just displaying two separate data values from the store connected to the grid.

{
    xtype: 'templatecolumn',
    tpl:   '{oneValue} {anotherVal}'
}
Action

The actioncolumn type we can display a set of icons that are going to execute an action when pressed. The actions are added to the items property.

{
    xtype: 'actioncolumn',
    items: [
        {
            icon: 'edit.gif',
            handler: function(grid, rowIndex, colIndex) {
              Ext.MessageBox.alert('Edit');
            }
        },
        {
            icon: 'delete.gif',
            handler: function(grid, rowIndex, colIndex) {
              Ext.MessageBox.alert('Delete');
            }
        }
    ]
}

If you want to have a column just displaying the row number for each row you can add it with:

columns: [
    Ext.create('Ext.grid.RowNumberer'),
    ...
]

The Data Package

The data package enables you to retrieve, decode and use data in your applications. It consists of 41 classes but the essential ones are Model, Store and Proxy.

Model

API Doc

A model represents some piece of data in the application. At its simplest a Model is just a set of fields and their data.

Models are used by Stores and registered via the ModelManager. Models can contain Associations and Validations. You can (should) also configure the Proxy directly on the model.

var User = Ext.regModel('User', {
    fields: ['id', 'name', 'email'],

    proxy: {
        type: 'rest',
        url : '/users',
        reader: {
            type: 'json',
            root: 'users'
        }
    }
});

var user = new User({name: 'Ed Spencer'});

//CREATE: calls the RestProxy's create function because the user has never been saved
user.save();

//UPDATE: calls the RestProxy's update function because it has been saved before
user.set('email', 'ed@sencha.com');

//DESTROY: calls the RestProxy's destroy function
user.destroy();

//READ: calls the RestProxy's read function
User.load(123, {
    success: function(user) {
        console.log(user);
    }
});

Model Associations

Models can be linked together with associations.

Stores

API Doc

The Store class encapsulates a client side cache of Model objects that can be bound to UI elements. Stores load data via a Proxy, and also provide functions for sorting, filtering and querying the model instances contained within it. The proxy can be configured on the store or preferably directly on the model object which it can use by being assigned a model.

Stores typically contain a subset of the server side data + local changes made to objects in the store, or the so called delta.

Store invokes the CRUD operations via its load and sync functions. Calling load uses the Proxy’s read operation, which sync utilizes one or more of create, update and destroy depending on the current Store delta. Stores support both local and remote sorting, filtering and grouping.

Proxies (Ext.data.proxy)

Proxies are used by stores to handle loading and saving of model data. All proxies implement the same interface and are at least in theory interchangeable.

Proxy
  Client
    Memory
    WebStorage
      LocalStorage
      SessionStorage
  Server
    Ajax
      Rest
    Direct
    JsonP

Ajax Proxy

Uses the browser´s XHR obj to perform generic AJAX requests.

JsonP Proxy

Used to overcome the “same origin policy”. This means it can retrieve data from another domain than it originated but the downside is that it can only return JavaScript instead of generic data. JsonP thus requires the return data to be wrapped in a global method call, passing the data in as the only parameter.

Memory Proxy

Loads data from a memory object

Direct Proxy

Allows the data store to interact with the Ext.direct remoting (RPC) providers.

REST Proxy

The REST Proxy is a subclass of the AJAX Proxy specialized to handle REST style resources.

LocalStorage and SessionStorage Proxies

Client side proxies that use the HTML5 webStorage API to persist data. LocalStorage for persisting data between sessions and SessionStorage is for persisting data throughout a single session.

Readers

Ext.data.reader

Readers are used by proxies to read data to a model or store. After a Proxy fetches the raw data, a Reader reads and parses it. A Reader is a class that takes the raw, unformatted data objects and abstracts the data points, known as dataIndexes, and arrange them into name/value pairs or model object instances.

Reader
  Array
  Json
  XML

Writer

Writer removes the requirement for the developer to have to code AJAX requests and exception handling

Themes

ExtJS uses SASS and Compass for creating themes.

The current themes reside under resources/themes

SASS and compass are Ruby gems. So in order to use them you need to have Ruby installed.

gem install compass

compass -v

Sencha SDK Tools

http://www.sencha.com/products/sdk-tools/

Creating a jsb3 file

After installing the Sencha SDK tools we can use them to generate a build (jsb3) file. From the JS application root folder enter something like:

luka:UI LuKa$ sencha create jsb -a ~/UI/index.html -p IFA.jsb3
  • -a The location of the html file containing scripts for build
  • -p Output path for .jsb3 file

Building

By default the created .jsb3 file will generate two “build files”. They are all-classes.js and app-all.js. They both consist of your app + references to all the ext.js classes needed for the application. They both contain the same thing but the first can be used for debugging while the other is compressed and meant to deploy in production. To generate these files you run the sencha build command:

luka:UI LuKa$ sencha build -p IFA.jsb3 -d ~/UI -v
  • -p path to the .jsb3 file to build
  • -d path to output directory
  • -v Verbose output from the build process

This should generate all-classes.js and app-all.js` to the chosen output folder. Copy app-all.js to UI folder in the VM and check in.

The deployed version of index.html should look something like:

<html>
<head>
    <title>IFA</title>
    <link rel="stylesheet" type="text/css" href="resources/css/ext-all-gray.css">
    <script type="text/javascript" src="extjs/ext.js"></script>
    <script type="text/javascript" src="app-all.js"></script>
</head>
<body>
    ....
</body>
</html>

JSBuilder

JSBuilder site

JSBuilder is an advanced JavaScript and CSS project build tool. It includes a tool for creating a dependencies file in the form of a JSB3-file. Navigate to the root folder of your application and run:

sencha create jsb -a index.html -p app.jsb3

or if it’s behind a running web server:

sencha create jsb -a http://myapp/index.html -p app.jsb3

This command scans the application and generates a .jsb3-file that we can use as a build file. To actually build you run:

sencha build -p app.jsb3 -d .

This command should output 2 files:

  • all-classes.js - This file contains all of your application’s classes unminified so it’s useful for debugging.

  • app-all.js - The minified build version of your application plus all of the Ext JS classes required to run it. This is the production-ready version of all classes .js and app.js combined.

You should make a separate .html-file for the production version and include ext.js + app-all.js.

Sencha RESTful API

To fully utilize the possibilities that the Sencha client side tools (ExtJS+Sencha Touch) offer here are some requirements on a Server API:

Reading Data

Create, Update & Delete

Sencha will behave differently depending on if you choose to communicate via the model or the store. To be able to use either-or on the client, the server should support handling a single object or a list of objects.

  • REST style CRUD API supporting list of items. A PUT or POST could include a list of several items like:

{“records”:[{“id”:6162315674,”owner”:”46992422@N08”,”secret”:”d94d1629f4”,”server”:6161,”farm”:7,”title”:”foo”}]}

  • Support for arbitrary AJAX methods that doesn’t conform to standard CRUD

  • Sencha Clients expects the server to return the record or records that were sent for manipulation. For example, if you create a new record, the API should, after a successful save, return that record as part of the results. If you modify several records and save them, the API should return all the modified records if they saved correctly. The reason for this is that it’s possible that your API will make additional changes to the records that should be reflected in your JavaScript code. Returning the full records ensures that your JavaScript application stays up-to-date with any changes made by your API.

  • Success or failure is indicated by returning an appropriate HTTP status code. If you’re returning a successful message, simply outputting JSON will return an acceptable status code. To indicate failure, you would return an error code in the 400 or 500 range. http://restpatterns.org/ HTTP_Status_Codes

You’ll need to look up how to send HTTP response headers in the documentation for your preferred API programming language.
